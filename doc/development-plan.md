# ReadingCoach 开发计划与排期建议

## 1. 文档目标

本文档用于把 ReadingCoach MVP 的开发步骤整理成可执行计划，帮助产品、iOS、后端和测试明确：

1. 先做什么
2. 哪些可以并行
3. 哪些必须串行
4. 每个阶段如何验收

## 2. 开发目标

MVP 阶段目标是跑通以下完整链路：

1. 用户登录
2. 拍照并裁剪文章区域
3. 上传图片并触发解析
4. 服务端完成 OCR、DeepSeek 解析、微软语音生成
5. 客户端逐句展示英文、翻译、讲解和音频
6. 用户第二天可复习昨天文章
7. 用户可在另一台设备登录后查看历史与学习进度

## 3. 团队角色建议

建议最少配置：

1. 产品/项目管理 1 人
2. iOS 开发 1 人
3. Go 后端开发 1 人
4. UI 设计 1 人
5. 测试可由开发兼任，后续补专职测试更稳

## 4. 总体开发顺序

推荐顺序：

1. 先冻结 MVP 范围与接口契约
2. 同时搭建 iOS 和 Go 基础工程
3. 后端先完成登录、上传、文章详情等基础接口
4. iOS 先完成登录、拍照裁剪、上传、列表和文章详情页骨架
5. 后端再完成 AI 异步链路
6. iOS 接入解析状态轮询和音频播放
7. 双方再联调学习记录、统计和同步
8. 最后完成稳定性优化、测试和上线准备

## 5. 并行与串行关系

### 5.1 可并行

1. UI 设计和后端骨架可并行
2. iOS 页面骨架和后端数据库设计可并行
3. iOS 上传页面与后端登录接口可并行
4. AI provider 接入与客户端历史页开发可并行

### 5.2 必须串行

1. 客户端联调登录前，后端必须先提供可用登录接口
2. 客户端联调上传前，后端必须先提供上传接口
3. 客户端展示真实文章详情前，后端必须先提供文章详情接口
4. 统计和同步功能依赖学习记录链路先跑通

## 6. 推荐排期

以下按 4 周 MVP 节奏编排，适合 1 名 iOS + 1 名后端的小团队。

## 6.1 第 1 周：基础工程与主链路骨架

### 产品与设计

1. 确认页面清单与核心流程
2. 确认登录方式优先级
3. 确认拍照裁剪交互稿
4. 确认学习页和历史页基础稿

### Go 后端

1. 初始化服务端仓库结构
2. 接入 Gin、PostgreSQL、Redis、Zap、JWT
3. 建立数据库迁移框架
4. 创建 `users`、`user_sessions`、`articles` 基础表
5. 完成 Apple 登录接口
6. 完成获取当前用户信息接口
7. 完成上传图片接口

### iOS 客户端

1. 初始化 SwiftUI 项目结构
2. 搭建底部 Tab 和页面路由
3. 完成登录页 UI
4. 接入 Apple 登录
5. 完成首页、我的页骨架
6. 完成拍照页和裁剪页基础交互

### 第 1 周验收标准

1. 用户可完成登录
2. 客户端可带 token 请求受保护接口
3. 客户端可拍照、裁剪，并把图片上传到服务端
4. 服务端能落库文章基础记录

## 6.2 第 2 周：解析链路与文章学习页

### Go 后端

1. 接入 Asynq
2. 实现 OCR provider
3. 实现 DeepSeek provider
4. 实现文章解析任务链路
5. 完成文章详情接口
6. 完成文章列表接口
7. 完成重新解析接口

### iOS 客户端

1. 完成上传后的处理中页面
2. 完成文章详情页 UI
3. 完成逐句卡片列表
4. 接入文章详情接口
5. 接入解析状态轮询
6. 完成历史列表页

### 联调重点

1. 上传成功后可创建解析任务
2. 客户端能看到处理中状态变化
3. 解析完成后能展示真实句子列表

### 第 2 周验收标准

1. 一篇文章从上传到解析完成的主链路跑通
2. 客户端能正确显示英文、翻译和拆句讲解
3. 历史列表能看到已学习文章

## 6.3 第 3 周：音频播放、学习记录、次日复习

### Go 后端

1. 接入微软语音 provider
2. 完成句子音频生成与对象存储上传
3. 完成学习记录接口
4. 完成文章学习进度接口
5. 完成昨日复习推荐接口

### iOS 客户端

1. 接入 AVFoundation 播放器
2. 完成单句播放与全文播放
3. 实现当前句高亮
4. 接入学习行为上报
5. 接入文章学习进度恢复
6. 接入昨日复习入口

### 联调重点

1. 音频能正常生成和播放
2. 当前播放句与返回进度一致
3. 学习完后第二天可进入复习模式

### 第 3 周验收标准

1. 逐句音频播放功能可用
2. 学习行为已落库
3. 复习入口能展示昨天文章

## 6.4 第 4 周：统计、同步、优化与测试

### Go 后端

1. 完成统计首页接口
2. 完成打卡日历接口
3. 完成同步启动接口
4. 增加错误处理、限流、重试和日志增强
5. 补关键单元测试和集成测试

### iOS 客户端

1. 完成统计页接入
2. 完成同步状态展示
3. 完成新设备登录后数据拉取
4. 完成错误态、空状态和网络异常处理
5. 完成体验优化和基础自测

### 联调重点

1. 新设备登录可看到历史和统计
2. 当日学习行为能反映到统计页
3. 解析失败、音频失败、网络失败都有合理提示

### 第 4 周验收标准

1. 登录、学习、复习、统计、同步全链路跑通
2. 主要异常场景可处理
3. MVP 具备可测试版本交付条件

## 7. 模块级开发步骤

## 7.1 后端优先级

1. `auth`
2. `article upload`
3. `article parse`
4. `article detail/list`
5. `tts`
6. `study`
7. `stats`
8. `sync`

## 7.2 iOS 优先级

1. 登录页
2. 首页骨架
3. 拍照页
4. 裁剪页
5. 上传处理中页
6. 文章学习页
7. 历史页
8. 统计页
9. 我的页与同步状态

## 8. 联调顺序

推荐按下面顺序联调：

1. `auth/apple/login`
2. `GET /me`
3. `POST /articles/upload`
4. `POST /articles/{id}/parse`
5. `GET /articles/{id}`
6. `GET /articles`
7. `POST /study/records`
8. `GET /articles/{id}/progress`
9. `GET /review/yesterday`
10. `GET /stats/dashboard`
11. `GET /sync/bootstrap`

## 9. 每阶段验收清单

## 9.1 登录阶段

1. 登录成功后 token 可用
2. 重新打开 App 可自动恢复登录态
3. 退出登录后受保护接口失效

## 9.2 上传与解析阶段

1. 客户端只上传裁剪后的图片
2. 服务端能收到图片并创建文章
3. 文章状态会从 `uploaded` 变为 `processing` 再到 `completed`

## 9.3 学习页阶段

1. 每句英文、翻译、讲解显示正确
2. 切换文章重新进入时数据一致
3. OCR 修正后二次解析可生效

## 9.4 音频阶段

1. 单句播放可用
2. 全文播放可用
3. 播放进度可恢复

## 9.5 统计与同步阶段

1. 今日学习篇数正确
2. 连续学习天数正确
3. 新设备登录后能看到历史和进度

## 10. 风险点与应对

## 10.1 Apple 登录调试复杂

应对：

1. 第 1 周先确保真机测试环境可用
2. 必要时先用验证码登录做开发兜底

## 10.2 OCR 和 DeepSeek 输出不稳定

应对：

1. 先用固定 mock 数据联调客户端
2. 服务端对模型输出做 JSON 校验和重试
3. 增加 OCR 文本编辑重解析能力

## 10.3 微软免费语音方案限制较多

应对：

1. 先保证句子级音频生成
2. 做缓存避免重复生成
3. 预留切换到正式商用 TTS 的 provider 接口

## 10.4 小团队联调阻塞

应对：

1. 后端优先提供 mock 接口
2. 客户端优先按 mock 数据完成页面
3. 每周至少安排 2 次联调检查

## 11. 推荐项目管理方式

建议拆成 4 个迭代看板：

1. 基础工程
2. 主链路开发
3. 联调与修复
4. 上线准备

任务卡建议至少包含：

1. 背景
2. 输入输出
3. 接口或页面
4. 验收标准
5. 是否阻塞其他任务

## 12. 开发开始前最后确认项

开始编码前建议确认以下事项：

1. 是否只支持 iPhone，不做 iPad 适配
2. Apple 登录是否作为唯一正式登录方式
3. 微软免费语音方案的接入路径是否已验证
4. OCR 供应商是否已最终确认
5. 对象存储和域名是否已准备
6. 开发环境和测试环境是否已建好

## 13. 结论

如果按照本文档执行，当前项目已经具备进入 MVP 开发的条件。

建议第一天就同时启动两条线：

1. Go 服务端骨架、登录和上传接口
2. iOS 登录、拍照裁剪和页面骨架

这样可以最快在第 1 周末看到第一条可联调链路。
